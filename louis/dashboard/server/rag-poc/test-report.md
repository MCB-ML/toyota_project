# TOPIC vs RAG 정답률/응답시간 테스트 보고서

**테스트 일시**: 2026-07-21
**테스트 방법**: `ktws_측정값_쿼리화.xlsx`(1-1/1-2시트)에서 실제 "GOLD 쿼리" 값이 채워진 15개
측정값("개체 값 이름") 전부를 자연어 질문으로 옮겨, TOPIC 분류 경로와 RAG(Pattern
Card/Fragment 10-Stage) 경로 양쪽에서 `--execute`로 실제 Fabric 실행까지 진행. GOLD 값은
같은 원본 GOLD SQL을 직접 실행해서 구했다(스키마 추정만 있고 GOLD 쿼리가 없는 fs11~fs21은
제외).

목록형(목록/breakdown) 케이스는 응답 SQL의 `TOP N`을 임시로 풀어 전체 행을 합산해
비교했다(채팅 표시용 100행 캡이 정답률 채점에 불리하게 작용하지 않도록).

## 요약

| | 정답률 | 평균 응답시간 |
|---|---|---|
| **TOPIC** | 8/15 (53.3%) | 6,653ms |
| **RAG** | 14/15 (93.3%) | 8,614ms |

- RAG가 정확도에서 크게 앞서지만, fragment별 개별 LLM 호출(케이스당 보통 1~5회) 때문에
  평균 응답시간은 TOPIC보다 약 30% 더 걸린다 — 정확도와 지연시간의 트레이드오프.
- RAG의 유일한 오답(#15, 계약 실적 목록)은 19,130 vs 19,132로 오차 2 — 세션 내내 관찰된
  Fabric 쪽 라이브 데이터(ETL 갱신) 변동성으로 추정되며, 재실행 시 값이 흔들리는 현상이 이
  프로젝트 전반에서 여러 차례 확인됨(구조적 버그로 보기 어려움).
- TOPIC의 오답 7건 중 다수(#2 당월목표, #5 영업기회 목표, #8 계약건수)는 이전부터 알려진
  라우팅 문서/생성 프롬프트의 한계이고, #3(별칭 불일치로 null)·#7(SQL 문법 오류)은 이번
  실행에서 새로 관찰된 비결정성 사례 — 실행할 때마다 TOPIC 경로의 실패 지점이 달라지는
  경향이 있다(같은 케이스가 이전 실행에선 통과했다가 이번엔 실패하기도 함).

## 이 테스트 전에 고친 버그 2건 (당일 세션)

1. **Fragment 간 날짜 불일치**: Stage 7이 fragment별로 독립 생성되다 보니 "이번달" 해석이
   fragment마다 달라지던 문제 → 결정적 파라미터 계산 단계(Stage 0.5)를 추가해 모든 fragment
   호출에 동일한 연월/월초/월말 값을 주입.
2. **없는 WHERE 조건 임의 추가** (#13, 영업기회 목표 목록에서 발견): 참고 SQL 템플릿에
   연월 조건이 없는데도 Stage 7이 "확정된 파라미터 값"의 연월을 임의로 추가해 92,185 대신
   11,466을 내던 문제 → "템플릿에 없는 조건은 추가하지 말 것"을 프롬프트에 명시해 해결,
   본 보고서의 #13 결과(92,185 ✅)로 재검증됨.

## 지식베이스 확장 (2026-07-21, 26 → 51개 Pattern Card)

`ktws_측정값_쿼리화.xlsx`가 갱신되면서(13개 시트, GOLD 쿼리 41건) 1-1/1-2시트 외에 6개
시트(1-1 시승 퍼널, 1-2 조직별 목록, 1-3 SC현황/근무일, 5-2 영업기회 목록, 6 핫보드,
7 SC 출고 매트릭스)에서 신규 GOLD 쿼리 25건을 추가로 반영해 `knowledgeBase/`를 확장했다
(`sql_sources` 26→51, `sql_fragments` 40→85, `business_rules` 10→13, `query_patterns` 26→51).

- **1-1시트**: 시승(test-drive) 퍼널 5건(전환율+건수 4변형) — 기존 KB에 없던 완전 신규 개념.
  계약 당월전체실적/목표/진행률 3건, 영업기회 당월전체실적 1건.
- **1-2시트**: 활동목표·기회전환률·기회진척률·계약전환률·계약진행률 조직별 목록 5건.
- **1-3시트**: **근무일(working day) 개념 전체가 신규**(진행/총/진행률 3건) + SC현황
  활동수·목표·진행률·전환율(딜러 슬라이서 없는 전사 합계 버전) 4건.
- **5-2시트**: 영업기회 상세 목록(리드 1건당 1행) 1건.
- **6시트**: 핫보드 미팅 상세 목록 1건 — 기존 `frag_hboard_hot_by_sc`(GOLD 미검증 추정치)와는
  형태가 달라 대체가 아니라 추가.
- **7시트**: SC 출고 매트릭스 2건 — 다른 항목과 비교가 안 될 만큼 복잡해서(각 6.7K/10.8K자,

  잘게 쪼갠 CTE 체인이 아니라 거의 원본 그대로의 단일 fragment로 반영.

### 검증 결과

25개 신규 패턴 전부 `inspectQuery.mjs --execute`로 Stage3 pattern 매칭 + Stage9b 라이브
Fabric 실행까지 확인 — 전부 `liveCheckPassed: true`. 기존 15개 GOLD 케이스 중 3개 재검증도
회귀 없음 확인(계약건수 280, 활동실적 목록 정상).

**발견하고 고친 버그 1건**: SC 출고 매트릭스(fs51, 상세표)가 Fabric 기본 요청 타임아웃
(mssql/tedious 기본값 15초)을 초과해 `Timeout: Request failed to complete in 15000ms`로
실패 — `server/fabricClient.js`의 커넥션 설정에 `requestTimeout: 60000`을 추가해 해결,
재검증 결과 정상 실행(50개 SC 상세행 + 합계행).

**알려진 한계 1건**: SC 출고 매트릭스 상세표는 합계(SC='합계') 행이 `sort_no=2147483647`로
항상 맨 끝에 오도록 정렬되는데, Stage8 Composer가 주입하는 `TOP 100`이 상세 SC 행이 99개를
넘으면 합계 행 도달 전에 결과를 잘라 합계 행이 누락될 수 있다(fragment 설명에 명시해둠).

**새로 생긴 애매성 1건**: 신규 "영업기회 건수(당월전체실적, coalesce)" 패턴이 기존
"영업기회 건수(당월활동실적)" 패턴과 같은 자연어 질문("영업기회 건수를 보여줘")에 둘 다
매칭될 수 있는 후보가 됐다 — 실제로 재검증 중 이전엔 활동실적 버전이 뽑히던 질문에 이번엔
coalesce 버전이 뽑혔다(1351 대신 1501). 둘 다 GOLD 쿼리 기반의 실제 존재하는 서로 다른
정의라 어느 쪽도 "틀린" 답은 아니지만, 사용자가 어느 정의를 원하는지 질문만으로는
모호할 수 있다는 점을 알아둘 것 — 필요하면 두 패턴의 `description`에 구분 문구를 더
명확히 추가하는 후속 작업을 고려.
